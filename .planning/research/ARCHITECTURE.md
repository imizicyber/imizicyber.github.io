# Architecture Patterns

**Domain:** Premium cybersecurity firm — Astro SSG static website
**Researched:** 2026-03-16
**Project:** Imizi Cyber (imizicyber.github.io)

---

## Current State Assessment

The existing architecture is sound and coherent. It follows SSG-first principles correctly: zero client JavaScript by default, content collections with Zod schema, a single data module (`src/data/site.ts`) as the source of truth, and a clean layout composition pattern. The gaps are not architectural violations — they are missing subsystems and quality-layer absences: no test infrastructure, inline scripts that mix concerns, hardcoded analytics IDs in source, no component index, and a monolithic `index.astro` that composes too many responsibilities in one file.

The enhancement milestone does not require an architectural rethink. It requires adding three new subsystems (analytics, test, performance) and refactoring the existing structure to enforce cleaner boundaries.

## Component Boundaries

### Layer 1: Data Layer (no dependencies)

**Location:** `src/data/`
**Files:** `site.ts`
**Responsibility:** Single source of truth for all site constants — brand, contact, social, analytics IDs.
**Communicates with:** Everything reads from it; nothing writes to it.
**Boundary rule:** No component, page, or layout should hardcode URLs, IDs, or brand strings. They must import from here.

**Target expansion:**

| File                   | Contents                                            | Replaces                                  |
| ---------------------- | --------------------------------------------------- | ----------------------------------------- |
| `src/data/site.ts`     | Brand, contact, social (existing)                   | No change                                 |
| `src/data/services.ts` | Service definitions (name, slug, description, CTAs) | Hardcoded repetition across service pages |
| `src/data/nav.ts`      | Navigation link definitions                         | Hardcoded nav items in Nav.astro          |

### Layer 2: Content Layer (depends on Data)

**Location:** `src/content/`
**Files:** `blog/*.mdx`, `content.config.ts`
**Responsibility:** Typed, validated content. All editorial content lives here, never in page files.
**Communicates with:** Build-time only. Pages consume via `getCollection()`. No runtime access.
**Boundary rule:** Pages must not inline large content. If it's editorial (blog posts, service descriptions, FAQ answers), it belongs in content collections or data files.

**Target expansion:**

| Collection             | Description                                        |
| ---------------------- | -------------------------------------------------- |
| `blog`                 | Blog posts — existing, maintain                    |
| _(future)_ `resources` | Downloadable resources if structured content grows |

### Layer 3: Component Layer (depends on Data, no circular deps)

**Location:** `src/components/`
**Responsibility:** Reusable UI elements with a single declared responsibility each.
**Communicates with:** Data layer (imports from `src/data/`). Components do NOT import other components directly — composition happens in layouts and pages.
**Boundary rule:** A component must not know about the page it lives in. It accepts props, renders HTML, optionally handles its own isolated client interaction.

**Component taxonomy (current + target):**

| Category            | Components                                                            | Notes                                              |
| ------------------- | --------------------------------------------------------------------- | -------------------------------------------------- |
| Structural          | `Nav.astro`, `Footer.astro`, `SkipLink.astro`                         | Navigation and page frame                          |
| SEO                 | `SEOHead.astro`, `SchemaOrg.astro`                                    | Head meta and structured data                      |
| Content display     | `BlogCard.astro`, `Breadcrumb.astro`, `Callout.astro`, `CtaBox.astro` | Rendering content items                            |
| Lead generation     | `ContactForm.astro`, `WhatsAppFloat.astro`                            | Conversion touchpoints                             |
| Consent & analytics | `CookieBanner.astro`                                                  | Consent gate for analytics loading                 |
| Interactive tools   | `ThemeToggle.astro`                                                   | Client-side UI state                               |
| _Target new_        | `BookingCTA.astro`                                                    | Reusable "Book a Consultation" CTA block           |
| _Target new_        | `ServiceCard.astro`                                                   | Service offering card for homepage grid            |
| _Target new_        | `CredentialBadge.astro`                                               | Founder cert/credential display unit               |
| _Target new_        | `TrustBar.astro`                                                      | Trust signals strip (stats, logos, certifications) |

**Boundary enforcement:** Each component gets a typed `Props` interface. No component should reach into global state except through its props or the explicit `site.ts` import.

### Layer 4: Layout Layer (depends on Components, Data)

**Location:** `src/layouts/`
**Responsibility:** Page structure wrappers. Define head, nav, footer, slot injection point. Compose components into full page frames.
**Communicates with:** Components (Nav, Footer, SEOHead, SchemaOrg, CookieBanner, WhatsAppFloat), Data (SITE constants for CSP, theme), Pages (via `<slot />`).

**Current layouts:**

| Layout                   | Purpose                                                                 | Used By                                |
| ------------------------ | ----------------------------------------------------------------------- | -------------------------------------- |
| `BaseLayout.astro`       | Root wrapper: head, nav, footer, CSP, analytics, cookie banner          | All pages                              |
| `BlogPostLayout.astro`   | Blog article frame: article element, auto-generated schemas, breadcrumb | Blog post pages                        |
| `SimplePageLayout.astro` | Light wrapper for legal/static pages                                    | Privacy policy, responsible disclosure |

**Target state:** No new layouts needed. `BaseLayout` may be split to separate concerns if it exceeds ~100 lines of logic. The CSP string construction belongs in a utility function, not inline in the layout.

### Layer 5: Page Layer (depends on all layers)

**Location:** `src/pages/`
**Responsibility:** Route definition and page-level composition. Pages import layouts, pass props, compose section components.
**Communicates with:** All layers.
**Boundary rule:** Pages should be thin composers. Sections longer than ~50 lines of markup that could be reused (hero, FAQ, process steps, CTA sections) should be extracted to components.

**Known violation:** `src/pages/index.astro` is a monolith. It contains the hero section, trust stats, why-us section, services grid, process steps, about/credentials section, blog feed, FAQ section, and contact form — all inline. This must be refactored into section components.

**Target page section components (extracted from index.astro):**

| Component                  | Section                         | Notes                            |
| -------------------------- | ------------------------------- | -------------------------------- |
| `HeroSection.astro`        | Homepage hero                   | Primary CTA placement            |
| `ServicesSection.astro`    | Services grid                   | Driven by `src/data/services.ts` |
| `ProcessSection.astro`     | How we work steps               |                                  |
| `CredentialsSection.astro` | Founder profile, certifications | Core credibility section         |
| `BlogFeedSection.astro`    | Latest blog posts preview       |                                  |
| `FAQSection.astro`         | FAQ accordion                   | Drives FAQPage schema            |
| `ContactSection.astro`     | Contact form wrapper            |                                  |

### Layer 6: Analytics & Consent Subsystem (cross-cutting, runtime only)

**Location:** `src/components/CookieBanner.astro` + future `src/lib/analytics.ts`
**Responsibility:** Gate all analytics behind explicit consent. Load GA4 on accept. Fire conversion events at lead generation moments.
**Communicates with:** `src/data/site.ts` (GA ID), localStorage (consent state), window.gtag API, external CDNs (googletagmanager.com).
**Boundary rule:** No analytics script runs without consent confirmation. Analytics IDs never appear in page markup — always sourced from `site.ts`.

**Target analytics event architecture:**

```
User action → Event function call → gtag() (if consented) → GA4
     |
     └→ Contact form submitted → gtag event: 'generate_lead'
     └→ Security quiz completed → gtag event: 'quiz_completed' (value: score)
     └→ PDF gate form submitted → gtag event: 'generate_lead' (source: 'quiz_pdf')
     └→ WhatsApp button clicked → gtag event: 'contact'
     └→ "Book a Consultation" CTA clicked → gtag event: 'cta_click'
```

**LinkedIn Insight Tag:** Loaded identically to GA4 — only on consent, injected dynamically. Add `linkedin.com` to CSP connect-src.

### Layer 7: Test Layer (build-time quality gate)

**Location:** `src/tests/` (unit) + `e2e/` (integration)
**Responsibility:** Prevent regressions. Validate component contracts. Test critical user paths (form submission, quiz flow, cookie consent, theme toggle).
**Communicates with:** Component source files only. Does not touch pages or layouts directly in unit tests.
**Boundary rule:** Tests must not make real network requests. Formspree and GA calls must be interceptable.

**Target test surface:**

| Test Type       | Tool                             | What It Covers                                                       |
| --------------- | -------------------------------- | -------------------------------------------------------------------- |
| Unit            | Vitest                           | Utility functions, schema validation logic, data transformation      |
| Component       | Vitest + Astro testing utilities | Props contracts, conditional rendering                               |
| Integration/E2E | Playwright                       | Cookie consent flow, form submission, quiz complete-to-PDF, nav, 404 |
| Build           | CI (GitHub Actions)              | `astro build` passes, no TypeScript errors, no linting errors        |

## Patterns to Follow

### Pattern 1: Props-first Components

Every component declares a typed `Props` interface. All external dependencies (content, config) are received as props. The component does not import from `../data/site.ts` unless it is a structural component (Nav, Footer, SEOHead) with a documented reason to do so.

```typescript
// Good: explicit props contract
interface Props {
  title: string;
  description: string;
  ctaHref: string;
  ctaLabel?: string;
}
```

### Pattern 2: Consent-Gated Analytics

All analytics calls are wrapped in a consent check. No analytics event fires without `localStorage.getItem('imizi-cookie-consent') === 'accepted'`. A shared utility function centralises this logic.

```typescript
// src/lib/analytics.ts
export function track(eventName: string, params?: Record<string, unknown>) {
  if (typeof window === 'undefined') return;
  if (localStorage.getItem('imizi-cookie-consent') !== 'accepted') return;
  if (typeof window.gtag !== 'function') return;
  window.gtag('event', eventName, params);
}
```

### Pattern 3: Build-Time Schema Generation

Structured data is always generated at build time. It never relies on client-side DOM construction. Each page passes a `schemas` array to `BaseLayout`, which passes it to `SchemaOrg` for render in `<head>`.

```astro
// Pattern: page defines schemas, layout renders them const schemas = [
buildOrganizationSchema(SITE), buildBreadcrumbSchema([...]) ];

<BaseLayout schemas={schemas} ... />
```

### Pattern 4: Thin Pages, Fat Components

Pages are composers. They import section components, pass data, and define the schema array. The visual complexity lives in components, not in `*.astro` page files.

```
src/pages/index.astro (~80 lines max after refactor)
  → imports HeroSection, ServicesSection, ProcessSection,
    CredentialsSection, BlogFeedSection, FAQSection, ContactSection
  → each section component is self-contained (~60-120 lines)
```

### Pattern 5: Environment Variables for Sensitive IDs

`ANALYTICS.gaId` and `ANALYTICS.formspreeId` in `site.ts` are not secrets (they are public client-side IDs by nature), but other integration tokens must use `import.meta.env` and never be committed. The pattern for any future integration:

```typescript
// src/data/site.ts — public IDs only, safe to commit
export const ANALYTICS = {
  gaId: 'G-XXXXXXXX', // public, safe
  linkedinPartnerId: '123456', // public, safe
} as const;

// .env (gitignored) — secrets only
FORMSPREE_SECRET_KEY = xxx;
```

### Pattern 6: Progressive Enhancement for Interactive Components

All interactive features (quiz, form, theme toggle) must have a functional HTML base state. JavaScript enhances but does not gate core functionality. For the quiz specifically, this means the calculate button should be accessible without JavaScript (fallback: standard form submission), even if the UX is degraded.

## Anti-Patterns to Avoid

### Anti-Pattern 1: innerHTML for User-Visible Content

**What:** Using `element.innerHTML = dynamicContent` with unsanitised strings (current quiz score/bullets construction).
**Why bad:** XSS vector if any input is ever user-controlled. Also breaks strict CSP.
**Instead:** Use `element.textContent` for text, `createElement` for DOM nodes, or Astro server-rendered HTML. For the quiz, all content is hardcoded in a JS array — safe today, but the pattern should be replaced with proper DOM construction.

### Anti-Pattern 2: Hardcoded Analytics IDs in Markup

**What:** Embedding `G-R7TC88KH9N` directly in page HTML or scripts.
**Why bad:** Makes rotation difficult, forces a full site rebuild to change, exposes config coupling.
**Instead:** All IDs sourced exclusively from `src/data/site.ts`, passed as props to components.

### Anti-Pattern 3: Monolithic Page Files

**What:** `index.astro` composing 500+ lines of HTML directly.
**Why bad:** Untestable, unreusable, hard to maintain, violates single-responsibility.
**Instead:** Extract each section into a named section component. Pages become < 100 lines.

### Anti-Pattern 4: CSP unsafe-inline for Scripts

**What:** Current CSP includes `script-src 'self' 'unsafe-inline'`.
**Why bad:** Defeats XSS protection entirely. Nonces or hashes are the correct mechanism.
**Instead:** For `is:inline` scripts that cannot be moved to external files (theme init FOUC prevention), use a nonce-based CSP. Astro 4+ supports nonce injection via middleware. For GitHub Pages (no server), a hash-based approach is the workaround.
**Note:** This is a moderate complexity change. Address in the security hardening phase, not the UI phase.

### Anti-Pattern 5: Loading External Scripts Without Fallbacks

**What:** Quiz PDF generation loads jsPDF from CDN without integrity checking.
**Why bad:** CDN compromise delivers arbitrary code to users. `crossorigin` + `integrity` attributes are required for external scripts.
**Instead:** Pin jsPDF to a specific version with a Subresource Integrity (SRI) hash. The existing fallback chain (cdnjs → unpkg → jsdelivr) is a good pattern; add `integrity` attributes.

### Anti-Pattern 6: Stateful Globals Without Boundaries

**What:** `window.handleCookieConsent`, `window.loadGA`, `window.resetCookieConsent` are global functions defined in an inline script.
**Why bad:** Pollutes global namespace, untestable, fragile if any script loads before CookieBanner.
**Instead:** Wrap in an IIFE or module pattern. For the reset function (needed from Privacy Policy page), use a custom event pattern: Privacy Policy page dispatches `CustomEvent('imizi:resetConsent')`, CookieBanner listens. No global function needed.

### Performance Subsystem (African Mobile Networks)

Mobile networks in sub-Saharan Africa commonly operate at 2G-3G speeds with high latency and intermittent connectivity. The architecture must address this explicitly.

**Critical performance rules:**

| Rule                               | Implementation                                                      | Boundary                                     |
| ---------------------------------- | ------------------------------------------------------------------- | -------------------------------------------- |
| Font loading non-blocking          | `rel="preload" onload` pattern (already in BaseLayout)              | Maintain                                     |
| No render-blocking scripts         | `is:inline` IIFE for theme init only; all else deferred             | BaseLayout, CookieBanner                     |
| Image optimization                 | All images use Astro `<Image />` component — produces WebP + srcset | Every img element                            |
| No framework JavaScript by default | Astro SSG zero-JS baseline                                          | No `client:load` directives unless essential |
| Third-party scripts deferred       | GA, LinkedIn load only on consent and asynchronously                | CookieBanner                                 |
| CSS budget                         | No unused utility classes; scoped component styles                  | global.css stays lean                        |
| Core Web Vitals targets            | LCP < 2.5s on 3G, CLS = 0, FID < 100ms                              | Measured via Lighthouse CI                   |

**Image architecture:**

```
All images → src/assets/images/
  → Astro <Image /> at build time
  → Outputs: WebP (primary) + JPEG fallback
  → Auto-generates srcset for responsive sizes
  → Adds width/height attributes (prevents CLS)
```

**Font strategy:**

- Plus Jakarta Sans: preload critical weights (600, 700) only; load 400/500/800 via `font-display: swap`
- JetBrains Mono: used only in code/badge elements — can be loaded lazily
- Self-hosting fonts (future): eliminates Google Fonts round-trip entirely for constrained networks

## Build Order Implications

The architecture has clear dependency layers that dictate build order:

```
Phase 1 must establish:
  → src/data/ extensions (services.ts, nav.ts)
  → Test infrastructure (Vitest + Playwright)
  → Linting + TypeScript strict mode
  Reason: All subsequent work depends on these contracts

Phase 2 can proceed:
  → Component refactor (extract section components from index.astro)
  → UI/UX redesign (component-level — no page-level breaking changes)
  Reason: Data layer and test infrastructure make this safe to refactor

Phase 3 can proceed:
  → Analytics subsystem (consent-gated events)
  → CookieBanner improvements (CustomEvent pattern, LinkedIn)
  Reason: Component contracts are now stable

Phase 4 can proceed:
  → SEO optimization (schema builders, keyword targeting)
  → Performance (image pipeline, font self-hosting)
  Reason: All structural components are stable; these are enhancement passes

Phase 5 can proceed:
  → New pages (founder credentials, updated service pages)
  → Lead generation funnel optimisation
  Reason: Infrastructure is complete; adding content is additive
```

**Critical dependency:** The test infrastructure must exist before major refactoring begins. Without tests, extracting section components from `index.astro` risks breaking the homepage with no regression detection.

**Critical dependency:** The `src/data/services.ts` data module must be built before `ServicesSection.astro` and updated service pages, or those components will have hardcoded content that must be refactored again.

## Component Communication Summary

```
site.ts ──────────────────────────────────────────────────────────→ All
content.config.ts + blog/*.mdx → getCollection() → Pages → Layouts
Pages → (props) → Layouts → (slot + props) → Components
CookieBanner → (consent event) → analytics.ts → window.gtag
ContactForm → (fetch) → Formspree API
ContactForm → (track event) → analytics.ts → window.gtag
QuizPage → (fetch) → Formspree API (gate form)
QuizPage → (dynamic load) → jsPDF CDN → client PDF
ThemeToggle → localStorage → <html data-theme>
BaseLayout → localStorage → ThemeToggle init (read only, FOUC prevention)
```

No circular dependencies exist in this model. Data flows in one direction: data layer → content layer → component layer → layout layer → page layer → browser.
