Title: "Penetration Test: East African Commercial Bank"

Sections (each with h2, following the article-body pattern from the pentest service page):

- **Client Description**: A BNR-regulated commercial bank operating in Rwanda and two neighbouring East African countries. 200+ employees, internet banking platform, mobile banking app, internal core banking system. The bank had never undergone an independent penetration test despite handling over $50M in daily transactions.

- **Engagement Type**: External and internal network penetration test + web application assessment of internet banking platform. 3-week engagement. Scope included external perimeter, internal network, internet banking web app, and API endpoints.

- **Key Findings** (use a numbered list with severity tags):
  1. CRITICAL: IDOR vulnerability in the internet banking API allowed any authenticated user to view account balances and transaction history of other customers by modifying account reference numbers in API requests.
  2. HIGH: JWT tokens issued by the authentication service had no expiry, allowing indefinite session reuse even after password changes.
  3. HIGH: Internal network segmentation was insufficient -- the penetration tester pivoted from a printer VLAN to the core banking database server using default SNMP community strings.
  4. MEDIUM: Administrative portal for loan processing was accessible from the public internet with default credentials on the staging environment.
  5. MEDIUM: Missing rate limiting on the OTP verification endpoint for mobile banking, enabling brute-force of 4-digit PINs.

- **Business Impact**: The IDOR vulnerability alone could have exposed the financial records of all 45,000+ retail banking customers. Combined with the JWT issue, an attacker could have maintained persistent access to customer accounts undetected. The network segmentation failure meant a compromise of any low-value asset (like a network printer) could lead to access to core banking data.

- **Recommendations**: Implement proper authorization checks on all API endpoints using server-side session context (not client-supplied identifiers). Deploy JWT with short-lived tokens (15min) and refresh token rotation. Segment the network into security zones with strict firewall rules between VLANs. Remove staging environments from public access. Implement rate limiting and account lockout on authentication endpoints.

- **Outcome**: All critical and high findings were remediated within 30 days. Free retest confirmed fixes. The bank now conducts quarterly penetration testing as part of their BNR compliance programme.

**Case study content -- realistic anonymised scenario for mobile money security in East Africa:**

Title: "Security Assessment: Mobile Money Platform"

Sections (same structure as case study 1):

- **Client Description**: A mobile network operator's fintech subsidiary providing mobile money services across three East African countries. The platform processed over 2 million transactions daily, serving both individual consumers and merchant payments. The company was preparing for a PCI DSS certification audit and needed an independent security assessment.

- **Engagement Type**: Comprehensive security assessment including mobile application testing (Android and iOS), USSD interface testing, API security review, and infrastructure assessment. 4-week engagement covering the full transaction lifecycle from customer registration through payment processing.

- **Key Findings** (numbered with severity):
  1. CRITICAL: USSD session fixation vulnerability allowed an attacker on the same mobile network to hijack active USSD sessions and authorize transactions on behalf of other users.
  2. CRITICAL: The mobile app stored the user's PIN in encrypted SharedPreferences, but the encryption key was hardcoded in the APK. Reverse engineering the app exposed the PIN of any user whose device was physically accessed.
  3. HIGH: The merchant payment API accepted negative transaction amounts, allowing a crafted request to reverse funds from merchant accounts to the attacker's wallet.
  4. HIGH: Certificate pinning was implemented in the Android app but not enforced on the iOS version, enabling man-in-the-middle interception of API traffic on iOS devices.
  5. MEDIUM: Agent registration API did not validate the national ID format, allowing registration with fabricated identities.
  6. LOW: Transaction notification SMS included full account numbers rather than masked versions, creating information disclosure risk on shared or stolen devices.

- **Business Impact**: The USSD session fixation could have enabled unauthorized fund transfers affecting any of the platform's 1.2 million active users. The hardcoded encryption key in the mobile app meant every customer's PIN was effectively stored in cleartext from a determined attacker's perspective. The negative transaction vulnerability could have been exploited for systematic fraud against merchants. These findings would have been blockers for PCI DSS certification.

- **Recommendations**: Implement server-side USSD session binding using device IMSI and timestamp validation. Move PIN storage to platform-provided secure enclaves (Android Keystore / iOS Keychain) with biometric binding. Add server-side transaction amount validation with strict positive-value enforcement. Enforce certificate pinning on all mobile platforms. Implement national ID format and checksum validation on registration endpoints. Mask account numbers in all customer-facing notifications.

- **Outcome**: The client prioritised the two critical findings for immediate remediation. All high and medium findings were addressed within 60 days. The security assessment report was submitted as evidence for their PCI DSS certification audit, which they passed on the subsequent attempt. The client engaged Imizi Cyber for ongoing quarterly assessments.
