<task type="auto">
  <name>Task 1: Wire GA4 events into ContactForm, WhatsAppFloat, and CTA buttons</name>
  <files>src/components/ContactForm.astro, src/components/WhatsAppFloat.astro, src/layouts/BaseLayout.astro</files>
  <action>
1. **ContactForm.astro** — Add a GA4 event on successful form submission:
   - Add a NEW `<script>` block (separate from the existing `define:vars` script since module imports cannot coexist with define:vars) that uses Astro's module script pattern: `<script>import { trackEvent } from '@/scripts/analytics'; ...</script>`
   - Listen for a custom DOM event `contact-form-success` on `document`.
   - In the existing inline script's success branch (where `r.ok` is true), after `form.reset()`, dispatch: `document.dispatchEvent(new CustomEvent('contact-form-success'))`.
   - In the new module script, listen: `document.addEventListener('contact-form-success', () => trackEvent('consultation_request', { form_location: window.location.pathname }))`.
   - This bridge pattern avoids mixing `define:vars` and ESM imports in the same script block.

2. **WhatsAppFloat.astro** — Add a GA4 event on click:
   - Add a `<script>` block at the bottom of the component.
   - Import `trackEvent` from `@/scripts/analytics`.
   - Select the `.wa-float` link and add a click listener that calls `trackEvent('whatsapp_click', { page: window.location.pathname })`.
   - The link still navigates normally (do NOT preventDefault).

3. **BaseLayout.astro** — Add CTA click tracking:
   - Add a `<script>` block (or append to an existing module script if one exists) that imports `trackEvent` from `@/scripts/analytics`.
   - Use event delegation on `document.body`: listen for clicks on elements matching `a[href*="contact"], a[href*="consultation"], .cta-btn, .hero-cta` selectors.
   - On match, call `trackEvent('cta_click', { cta_text: element.textContent?.trim() || '', page: window.location.pathname })`.
   - IMPORTANT: Do NOT preventDefault — CTAs must still navigate.

Note on ANLT-03 (resource download tracking): The resources page (`/resources/`) does not currently have trackable download links with a clear download mechanism. Add a comment in analytics.ts noting that resource download tracking (`resource_download` event) should be wired when the resource download feature is built in Phase 9 (LEAD-03). This keeps ANLT-03 acknowledged without implementing dead code.
</action>

  <name>Task 2: E2E tests for consent gating and consent persistence</name>
  <files>e2e/analytics.spec.ts, e2e/cookie-banner.spec.ts</files>
  <action>
1. **Create `e2e/analytics.spec.ts`** with these tests:

- **"no analytics requests before consent"**: Navigate to `/` with a fresh context (no localStorage). Set up `page.route('**/*googletagmanager.com*', ...)` and `page.route('**/*snap.licdn.com*', ...)` to intercept and track requests. Wait for page load and the cookie banner to be visible. Assert that zero requests were made to either analytics domain. This directly validates ANLT-07.

- **"analytics load after accepting consent"**: Same route interception setup. Navigate to `/`, click `#cookie-accept`, then wait up to 5 seconds for a request to `googletagmanager.com`. Assert at least one GA request was intercepted after acceptance.

- **"rejecting cookies prevents analytics loading"**: Navigate, click `#cookie-reject`, wait 2 seconds, assert zero analytics requests to either domain.

2. **Extend `e2e/cookie-banner.spec.ts`** with:
   - **"consent persists across page reload"**: Navigate to `/`, accept cookies, reload the page. Assert the cookie banner is NOT visible (hidden or not in DOM). This validates that returning visitors are not prompted again.

   - **"consent persists across page navigation"**: Accept cookies on `/`, navigate to `/blog/`. Assert banner is not visible.

All tests should use `test.describe` grouping. Use Playwright's `page.route()` for network interception (NOT `page.on('request')` which is less reliable for this pattern).
</action>
